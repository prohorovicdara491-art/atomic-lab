4. Ответы на контрольные вопросы
Вопрос 1: Как использование git add -p помогает при отладке через git bisect?
Ответ:

git add -p (интерактивное добавление изменений) позволяет разбить большой набор изменений на несколько атомарных коммитов, где каждый коммит содержит только одно логическое изменение.

Как это помогает git bisect:

git bisect — это инструмент для поиска коммита, который ввел ошибку. Он работает по принципу бинарного поиска: делит историю пополам и проверяет, есть ли ошибка.

С атомарными коммитами:
git bisect точно указывает на конкретный коммит, вызвавший ошибку. Например, если баг появился после добавления новой функции, bisect покажет коммит feat: add user status function.

Без атомарности:
Коммит содержит несколько изменений (рефакторинг + новую функцию + исправление). Bisect укажет на этот коммит, но непонятно, какое именно изменение вызвало проблему. Придется тратить дополнительное время на анализ.

Пример из практики:

bash
# Атомарные коммиты
e7488e5 refactor: rename variables          # Ошибки нет
9c74a00 feat: add user status function      # Ошибка появилась здесь!
fa41d41 fix: add error handling             # Ошибка уже есть

# git bisect сразу указывает на 9c74a00
Вопрос 2: Почему наличие requirements.lock.txt критично для командной работы и чем оно лучше обычного requirements.txt с версиями без минорных номеров?
Ответ:

requirements.lock.txt фиксирует точные версии всех зависимостей, включая транзитивные (зависимости зависимостей).

Сравнение подходов:

Тип файла	Пример	Риски
Без версий	requests	Установится последняя версия (может сломать код)
С версиями	requests==2.31.0	Фиксирует только одну библиотеку, но не её зависимости
lock-файл	requests==2.31.0
urllib3==2.0.4
certifi==2023.7.22	Фиксирует всё — полная воспроизводимость
Почему это критично для командной работы:

Проблема	Решение с lock-файлом
У разработчика A версия numpy 1.24, у B — 1.25 (код падает)	Все используют одинаковые версии
На CI/CD сервере установилась новая версия библиотеки (сломала сборку)	Версии фиксированы — сборка предсказуема
Через полгода нужно восстановить окружение	lock-файл позволяет точно воспроизвести
Обновилась транзитивная зависимость (например, urllib3)	Транзитивные зависимости тоже зафиксированы
Пример непредсказуемого поведения:

python
# Без lock-файла
# Сегодня: requests==2.31.0, urllib3==2.0.4 — код работает
# Завтра: requests==2.32.0, urllib3==2.1.0 — код падает

# С lock-файлом
# Всегда одни и те же версии — код работает стабильно
Вопрос 3: В чем преимущество Makefile перед текстовой инструкцией в README?
Ответ:

Makefile — это исполняемая документация. Вместо того чтобы писать в README "выполните команду 1, потом команду 2, потом команду 3", вы создаете цели, которые автоматизируют эти процессы.

Сравнение:

Аспект	Текстовая инструкция в README	Makefile
Выполнение	Нужно вручную вводить каждую команду	Достаточно одной команды: make install
Ошибки	Можно ошибиться при вводе	Команды выполняются автоматически
Единообразие	Каждый разработчик может ввести по-разному	Все используют одинаковые команды
Зависимости	Нужно помнить порядок	Makefile сам определяет порядок
Документация	Только текст	make help показывает все команды
Пример Makefile:

makefile
.PHONY: install test run

install:
	python -m venv venv
	venv/bin/pip install -r requirements.lock.txt

test:
	venv/bin/pytest test_app.py -v

run:
	venv/bin/python app.py
Использование:

bash
make install   # вместо трех команд
make test      # вместо pytest test_app.py -v
make run       # вместо python app.py
Вопрос 4: Как тесты реализуют принцип «живой документации» и почему фиксация seed важна для воспроизводимости?
Ответ:

Тесты как живая документация:

Тесты в стиле Given-When-Then служат документацией, которая всегда актуальна (в отличие от обычной документации, которая может устареть).

python
def test_user_status():
    # GIVEN (Дано) — подготовка данных
    user = {"is_active": True}
    
    # WHEN (Когда) — вызов тестируемой функции
    status = get_user_status(user)
    
    # THEN (Тогда) — проверка результата
    assert status == "active"
Что показывает этот тест:

Как подготовить данные — нужен словарь с ключом is_active

Как вызвать функцию — get_user_status(user)

Что ожидать — для активного пользователя возвращается "active"

Почему фиксация seed важна:

random.seed(42) делает случайные числа детерминированными:

python
# Без seed
random.random()  # 0.123456 (разный при каждом запуске)
random.random()  # 0.789012

# С seed
random.seed(42)
random.random()  # 0.639426 (всегда одинаковый)
random.random()  # 0.025010 (всегда одинаковый)
Последствия без seed:

Тесты могут иногда проходить, иногда падать (flaky tests)

Невозможно воспроизвести ошибку, которая произошла один раз

CI/CD может показывать разные результаты при каждом запуске

С seed:

Тесты всегда дают одинаковый результат

Ошибки воспроизводятся стабильно

CI/CD предсказуем
